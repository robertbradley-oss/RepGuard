# Phase 4.21 OpenAI Vision Sandbox Schema Plan

Date: 2026-06-01

Primary agent role: Integration Readiness Agent

Supporting reviews: Product Strategy, Architecture and Maintainability, Receipt Intelligence, Scoring and Safety, Privacy and Evidence Safety, QA Harness, Deployment and Release

## 1. Purpose And Scope

Phase 4.21 is an OpenAI Vision sandbox schema planning-only milestone. ClaimGuard needs a schema plan before any OpenAI Vision-style sandbox implementation because the future schema will define what a provider-like output may contain, what it must omit, how unsupported and failure states are represented, how privacy and retention rules are made visible, and how altered-or-AI-generated-image uncertainty remains a review signal rather than a conclusion.

This milestone is planning-only. It does not implement runtime TypeScript schema or types. It does not add OpenAI SDKs, provider SDKs, environment variables, provider calls, provider implementation, live OCR, real uploads, multipart parsing, binary parsing, storage, persistence, UI, route behavior changes, real evidence processing, production analysis, or live AI/photo analysis. It does not wire `ClaimReviewWorkflow`, does not route `ProductPhotoReviewPanel`, does not change `analyzeEvidenceFile`, does not change `LocalAnalysisResult`, does not change receipt parser/scoring/report behavior, does not change existing `POST /api/analysis/ocr` behavior, does not change existing `POST /api/analysis/mock-provider` behavior, does not deploy, and does not start Phase 4.22.

This is not live AI. It is not production analysis. It is a schema-planning milestone only, not a provider-integration milestone.

## Current Baseline

Phase 4.21 inherits these approved boundaries:

- Phase 4.9 recommends OCR-specialized extraction first and OpenAI Vision-style reasoning later for visual context, screenshots/layout, product-photo context, and altered-or-AI-generated-image uncertainty signals.
- Phase 4.10 plans provider abstraction concepts for provider mode, evidence types, privacy/redaction, cost/timeout, failure normalization, and review-support outputs.
- Phase 4.11 through Phase 4.18 establish mock-before-live as the implementation path and keep mock OCR/mock vision synthetic-only.
- Phase 4.19 plans a future OpenAI Vision-style sandbox as developer-only, synthetic/anonymized-fixture-only, privacy-safe, review-signal-only, and separate from live receipt analysis.
- Phase 4.20 plans prompt families and output contract language before any sandbox implementation.
- Existing `POST /api/analysis/ocr` remains exact `fixtureKey` JSON-only synthetic fixture route behavior.
- Existing `POST /api/analysis/mock-provider` remains synthetic/mock-only and adapter-only.
- The mock provider adapter remains the test boundary before live provider behavior.
- `analyzeEvidenceFile` and `LocalAnalysisResult` remain unchanged.
- Receipt behavior remains unchanged.

## 2. Future Top-Level Schema Identity

The future conceptual schema should be identified as an OpenAI Vision-style sandbox output, not a live provider result, not a receipt result, and not a claim disposition.

Future top-level identity fields:

- `schemaVersion`: required. Suggested first value: `openai-vision-sandbox-schema/v1`.
- `providerMode`: required. Allowed value for this schema plan: `sandbox`.
- `providerFamily`: required. Allowed value for this schema plan: `openai-vision-style`.
- `analysisMode`: required. Describes the prompt/output family.
- `evidenceType`: required. Broad evidence type for the sandbox fixture.
- `fixtureScope`: required. Describes the approved fixture class.
- `resultStatus`: required. Describes success, unsupported, refusal, provider failure, cost limit, validation failure, or internal sandbox error.
- `generatedAt`: required for future runtime output, generated by ClaimGuard sandbox code rather than copied from customer evidence. In planning, use only as an operational timestamp concept.
- `requestId`: required for future sandbox tracing, generated by ClaimGuard sandbox code and safe to log only if it is not a provider payload, storage handle, customer identifier, case identifier, claim identifier, ticket identifier, or evidence identifier.

Allowed conceptual `analysisMode` values:

- `visual-context-review`
- `receipt-visual-review`
- `order-screenshot-review`
- `product-photo-review`
- `damaged-product-review`
- `altered-ai-uncertainty-review`
- `mixed-evidence-review`
- `unsupported-evidence-review`

Allowed `fixtureScope` values:

- `synthetic-fixture`
- `synthetic-mock`
- `redacted-anonymized-fixture-later-approved`

Allowed `resultStatus` values:

- `completed`
- `unsupported`
- `refused`
- `provider-timeout`
- `provider-rate-limited`
- `provider-unavailable`
- `cost-limit-reached`
- `schema-validation-failed`
- `internal-sandbox-error`

Required schema identity validation rules:

- `providerMode` must be locked to `sandbox`.
- `providerFamily` must be locked to `openai-vision-style`.
- `resultStatus` must be one of the allowed values.
- `analysisMode` must be one of the allowed values.
- No identity field may contain provider payload identifiers, storage handles, URLs, local paths, customer identifiers, or production case identifiers.

## 3. Future Evidence Classification Fields

The schema should describe evidence type without exposing real identifiers or making unsupported evidence sound suspicious.

Future evidence classification fields:

- `evidenceType`: required. Broad evidence type, such as receipt image, order screenshot, product photo, damaged-product photo, mixed evidence, or unsupported evidence.
- `evidenceSubtype`: optional. Narrower fixture description, such as cropped synthetic receipt or synthetic close-up product photo.
- `evidenceCategory`: required. Must use an allowed category value.
- `fixtureKey`: required for synthetic fixture outputs, but must be an allowlisted synthetic fixture key only.
- `fixtureDescription`: required for completed synthetic fixtures; optional for failure states. Must be generic and privacy-safe.
- `inputRedactionStatus`: required. Examples: `synthetic-not-applicable`, `redacted`, `not-applicable-unsupported`, `blocked-redaction-required`.
- `inputIdentifierPolicy`: required. Must state whether identifiers were omitted, redacted, disallowed, or not applicable.
- `sourceRestriction`: required. Must state that the output is sandbox-only and not production analysis.

Allowed evidence categories:

- synthetic receipt image
- synthetic order screenshot
- synthetic product photo
- synthetic damaged-product photo
- mixed synthetic evidence
- unsupported synthetic fixture
- later-approved redacted anonymized fixture

Disallowed evidence categories:

- real customer receipt
- real customer product photo
- real ticket attachment
- unredacted customer evidence
- production upload object
- public image URL
- storage object handle
- provider payload replay

Field rules:

- `evidenceCategory` is required for every result status.
- `fixtureKey` is required for `completed` and `unsupported` synthetic fixture outputs.
- `fixtureKey` should be omitted or null for provider failure outputs where no fixture was evaluated.
- `fixtureDescription` must not include filenames, local paths, URLs, case IDs, claim IDs, ticket IDs, order numbers, names, emails, phone numbers, addresses, payment values, raw metadata, private background details, or provider payload details.
- `sourceRestriction` must make clear that this is sandbox output only.

## 4. Future Visual Summary Fields

The schema should support safe visual/context summaries while avoiding raw OCR reproduction, accusations, and claim decisions.

Future visual summary fields:

- `visualSummary`: required for `completed`; optional or null for failure/refusal states.
- `visibleContextSummary`: optional for evidence where context is visible and privacy-safe.
- `documentOrPhotoContext`: optional. Describes whether the fixture appears document-like, screenshot-like, product-photo-like, damaged-product-like, mixed, or unsupported.
- `layoutObservations`: optional list of privacy-safe layout notes.
- `qualityObservations`: optional list of image quality notes.
- `contentBoundaries`: required for `completed`; optional for failure states. Describes what the sandbox did not assess.

Rules:

- Summaries must avoid private identifiers.
- Summaries must avoid raw OCR reproduction.
- Summaries must not accuse the customer.
- Summaries must not decide the claim.
- Summaries must not say fraud, fake, forged, or altered is confirmed.
- Summaries must be written for internal reviewer context, not customer-facing conclusion output.

## 5. Future Observation Fields

Observations are structured review-relevant facts or context notes. They are not conclusions, decisions, proof, or uncertainty scores.

Each future observation object should include:

- `observationId`: required. Stable within one output, such as `obs-001`.
- `category`: required. Must use an allowed category.
- `description`: required. Privacy-safe, review-relevant observation.
- `basis`: required. Describes the visible basis at a high level without raw identifiers.
- `confidenceLevel`: required. Allowed values should align with confidence note levels: `low`, `medium`, `high`, or `not-applicable`.
- `relatedRegionDescription`: optional. Generic region description only, such as upper receipt area or lower-right product surface.
- `privacySafe`: required boolean. Must be true for observations included in output.
- `manualReviewRelevance`: required. Explains why a reviewer may care.

Allowed observation categories:

- `layout`
- `visual-consistency`
- `image-quality`
- `document-structure`
- `product-context`
- `damage-context`
- `screenshot-context`
- `receipt-context`
- `limitation`
- `unsupported`

Rules:

- Observations are not conclusions.
- Observations must remain separate from uncertainty signals.
- Observations must not include private identifiers.
- Observations must be phrased as review-relevant context.
- Observation IDs may be referenced by uncertainty signals, manual-review drivers, confidence notes, and altered/AI uncertainty fields.
- Unsupported/refusal/failure states may include only limitation-style observations when useful and safe.

## 6. Future Uncertainty Signal Fields

Uncertainty signals are review signals derived from observations or limitations. They should never become wrongdoing findings, support dispositions, or customer accusations.

Each future uncertainty signal object should include:

- `signalId`: required. Stable within one output, such as `sig-001`.
- `signalType`: required. Must use an allowed signal type.
- `level`: required. Suggested allowed values: `low`, `medium`, `high`, `not-applicable`.
- `description`: required. Review-signal wording only.
- `supportingObservationIds`: required when observations exist; empty only when the signal is based on provider output limitation or unsupported state.
- `confidenceLevel`: required. Analysis reliability, not claim truth.
- `manualReviewDriver`: required boolean.
- `safeLanguage`: required. Must include a review-only, non-accusatory phrasing note.

Allowed signal types:

- `visual-inconsistency-uncertainty`
- `layout-structure-uncertainty`
- `screenshot-context-uncertainty`
- `product-photo-context-uncertainty`
- `damage-context-uncertainty`
- `altered-or-ai-generated-image-uncertainty`
- `unsupported-evidence-limitation`
- `provider-output-limitation`

Rules:

- Signals are review signals only.
- Signals must not become fraud findings.
- Signals must not become final claim decisions.
- Signals must link back to observations where possible.
- Signals must not include customer-facing accusations.
- Signals must not recommend denial, approval, rejection, or refund.
- Unsupported and provider failure signals must be limitations only.

## 7. Altered/AI-Generated-Image Uncertainty Schema

The future 1-100 field should use the field group name `alteredOrAiGeneratedImageUncertainty`.

Future field group:

- `value`: integer 1-100 only when applicable. Must be null or omitted when not applicable, unsupported, refused, timed out, rate-limited, unavailable, cost-limited, schema-invalid, or internally errored.
- `scale`: required when `value` is present. Allowed value: `1-100`.
- `meaning`: required when the field group is present. Must say altered-or-AI-generated-image uncertainty, review signal only, manual-review driver, not proof, and not a final decision.
- `applicability`: required. Suggested allowed values: `applicable`, `not-applicable`, `unsupported`, `unavailable`, `refused`, `schema-invalid`.
- `confidenceLevel`: required. Analysis reliability only.
- `supportingObservationIds`: required when applicable and observations exist.
- `limitations`: required. Must explain why the value is incomplete and review-only.
- `manualReviewDriver`: required boolean.
- `safeLanguage`: required. Must use review-only wording and avoid accusations.

Rules:

- `value` is 1-100 only when applicable.
- `value` must be nullable or omitted when not applicable.
- `meaning` must say uncertainty/review signal only.
- The value must not represent proof of alteration, proof of AI generation, proof of fraud, final claim decision, deny/refund recommendation, or customer accusation.
- The field must not be named "fake score," "fraud score," "AI-generated confidence," or "forgery score."
- Required wording: "altered-or-AI-generated-image uncertainty," "review signal only," "manual-review driver," "not proof," "not a final decision."
- A high value should increase reviewer attention only.
- A low value must not say the image is genuine, verified, unaltered, or safe to approve.

## 8. Confidence Notes

Confidence notes describe analysis reliability. They do not describe whether a claim is true.

Each future confidence note should include:

- `confidenceId`: required.
- `area`: required. Examples: visual summary reliability, observation reliability, altered/AI uncertainty applicability, provider output availability, evidence type fit.
- `level`: required. Must use an allowed level.
- `reason`: required. Explains why reliability is limited or stronger.
- `limits`: required. Explicitly states what confidence does not mean.
- `recommendedReviewAction`: required. Reviewer action only.

Allowed levels:

- `low`
- `medium`
- `high`
- `not-applicable`

Rules:

- Confidence is about analysis reliability, not claim truth.
- Confidence must not be treated as fraud probability.
- Low confidence should drive manual review, not automatic denial.
- High confidence must not imply proof, external verification, customer wrongdoing, or final claim outcome.

## 9. Manual-Review Drivers

Manual-review drivers are internal reviewer aids. They should guide review posture without creating an automated disposition.

Each future manual-review driver should include:

- `driverId`: required.
- `priority`: required. Must use an allowed priority.
- `reason`: required. Non-accusatory, internal review reason.
- `relatedObservationIds`: optional list.
- `relatedSignalIds`: optional list.
- `recommendedInternalAction`: required. Suggested actions include human review, request clearer eligible evidence, compare with approved OCR result, or operational follow-up.
- `customerSafeBoundary`: required. Explains what should not be said to the customer as a conclusion.

Allowed priorities:

- `low`
- `medium`
- `high`

Rules:

- Drivers are internal review aids only.
- Drivers must not be customer accusations.
- Drivers must not recommend automatic denial/refund.
- Drivers should suggest human review, additional evidence request, or operational follow-up only.

## 10. Limitations And Safe Support Summary

Limitations should be first-class schema objects so output cannot overclaim.

Future summary fields:

- `limitations`: required for all result statuses.
- `safeSupportSummary`: required for all statuses. Safe internal support summary, not a customer conclusion.
- `internalReviewSummary`: required for `completed`; optional for unsupported/failure states.
- `customerSafeSummary`: required for all statuses if any summary would later be shown or adapted for customer-facing language.

Each limitation should include:

- `limitationId`: required.
- `type`: required. Must use an allowed limitation type.
- `description`: required.
- `impact`: required. Explains what the limitation prevents.
- `recommendedHandling`: required. Manual-review or operational handling only.

Allowed limitation types:

- `image-quality`
- `evidence-type`
- `missing-context`
- `provider-boundary`
- `privacy-redaction`
- `unsupported-evidence`
- `timeout`
- `rate-limit`
- `cost-limit`
- `schema-validation`
- `sandbox-only`

Rules:

- Limitations must be clear and must not overclaim.
- Customer-safe summaries must avoid blame language.
- Internal summaries may discuss review concerns but must still avoid proof/fraud confirmation.
- Unsupported and provider failure states must include at least one limitation.
- Completed outputs must include `sandbox-only` and no-external-verification limitations.

Privacy and retention metadata should be required so sandbox output cannot silently imply that raw evidence, provider payloads, or identifiers are retained.

Future privacy/retention metadata fields:

- `privacyFlags`: required.
- `retentionFlags`: required.
- `redactionStatus`: required.
- `identifierHandling`: required.
- `payloadLoggingPolicy`: required.
- `providerPayloadRetention`: required.
- `objectUrlPolicy`: required.
- `storagePolicy`: required.

Suggested `privacyFlags` fields:

- `syntheticOnly`
- `realCustomerEvidenceAllowed`
- `realCustomerEvidenceUsed`
- `customerIdentifiersAllowed`
- `customerIdentifiersReturned`
- `rawOcrReturned`
- `providerPayloadReturned`
- `publicImageUrlAllowed`
- `storageHandleAllowed`

Suggested `retentionFlags` fields:

- `fileRetained`
- `rawOcrRetained`
- `providerPayloadRetained`
- `providerPayloadLogged`
- `objectUrlRetained`
- `storageUsed`
- `persistenceUsed`

Rules:

- Real customer evidence is not allowed by default.
- Raw provider payload logging is not allowed by default.
- Raw OCR retention is not allowed by default.
- Object URL retention is not allowed.
- Storage/persistence requires a separate approved milestone.
- Redaction is required before later anonymized fixture testing.
- Future schema validation should fail if any default-off privacy or retention flag is true without a separately approved milestone.

## 12. Cost And Timeout Metadata

Cost and timeout metadata are operational only. They must not become evidence-risk or customer-risk signals.

Future `costMetadata` fields:

- `estimatedInputUnits`: required when a provider operation is attempted; zero or null for blocked-before-provider states.
- `estimatedOutputUnits`: required when a provider operation is attempted; zero or null for blocked-before-provider states.
- `estimatedCostBucket`: required. Suggested values: `none`, `low`, `medium`, `high`, `unknown`, `blocked-by-limit`.
- `costLimitApplied`: required boolean.

Future `timeoutMetadata` fields:

- `timeoutMs`: required when provider operation is attempted.
- `timedOut`: required boolean.
- `retryAttempted`: required boolean.
- `retryPolicy`: required. Suggested value for first sandbox implementation: `no-automatic-retry`.

Rules:

- No automatic retry by default.
- Provider timeout is operational only.
- Provider cost limit is operational only.
- Provider failure must not imply customer risk.
- Cost metadata must not expose provider payloads.
- Timeout metadata must not expose provider request IDs, response IDs, stack traces, URLs, or dashboard links.

## 13. Failure And Unsupported Shapes

Every future result status needs a predictable shape so callers do not infer conclusions from missing fields.

### Completed Result

Required fields: top-level identity, evidence classification, visual summary fields, observations, uncertainty signals, confidence notes, manual-review drivers, limitations, safe summaries, privacy flags, retention flags, cost metadata, timeout metadata, schema validation status.

Prohibited fields: raw OCR text, provider payloads, object URLs, storage handles, public image URLs, customer identifiers, claim disposition, denial/refund recommendation, receipt scoring output, `LocalAnalysisResult`.

Allowed limitations: `sandbox-only`, `provider-boundary`, `privacy-redaction`, `image-quality`, `missing-context`, `evidence-type`.

Allowed safe summary: review-support summary only.

Uncertainty signals allowed: yes, when linked to observations or limitations.

Altered/AI uncertainty allowed: yes, only when applicable.

Manual-review drivers allowed: yes.

Required customer-safe boundaries: no proof, no accusation, no final claim decision.

### Unsupported Evidence Result

Required fields: top-level identity, evidence classification, `resultStatus: unsupported`, limitations, safe summaries, privacy flags, retention flags, schema validation status.

Prohibited fields: altered/AI uncertainty `value`, proof language, claim disposition, raw identifiers, raw provider payloads.

Allowed limitations: `unsupported-evidence`, `evidence-type`, `sandbox-only`, `missing-context`.

Allowed safe summary: unsupported evidence is an evidence limitation only.

Uncertainty signals allowed: only `unsupported-evidence-limitation`.

Altered/AI uncertainty allowed: field group may be present only with `applicability: unsupported` and no `value`.

Manual-review drivers allowed: yes, for request eligible evidence or reviewer inspection.

Required customer-safe boundaries: unsupported does not mean suspicious and does not decide a claim.

### Prompt Refusal Result

Required fields: top-level identity, `resultStatus: refused`, limitations, safe summaries, privacy flags, retention flags, schema validation status.

Prohibited fields: observations that sound like analysis, uncertainty `value`, customer-risk language, provider raw refusal text.

Allowed limitations: `provider-boundary`, `sandbox-only`, `unsupported-evidence` when applicable.

Allowed safe summary: sandbox could not produce eligible output under the approved contract.

Uncertainty signals allowed: only `provider-output-limitation`.

Altered/AI uncertainty allowed: field group may be present only with `applicability: refused` and no `value`.

Manual-review drivers allowed: yes, because approved automated sandbox output is unavailable.

Required customer-safe boundaries: refusal is not evidence about the customer or claim.

### Provider Timeout Result

Required fields: top-level identity, `resultStatus: provider-timeout`, timeout metadata, limitations, safe summaries, privacy flags, retention flags, schema validation status.

Prohibited fields: completed observations, uncertainty `value`, provider stack traces, provider request IDs, raw payloads.

Allowed limitations: `timeout`, `provider-boundary`, `sandbox-only`.

Allowed safe summary: provider timeout; sandbox output unavailable.

Uncertainty signals allowed: only `provider-output-limitation`.

Altered/AI uncertainty allowed: field group may be present only with `applicability: unavailable` and no `value`.

Manual-review drivers allowed: yes, because output is unavailable.

Required customer-safe boundaries: timeout is operational only.

Prohibited fields: completed observations, uncertainty `value`, raw provider error payloads, URLs, provider trace details.

Allowed safe summary: provider unavailable; operational limitation only.

Uncertainty signals allowed: only `provider-output-limitation`.
